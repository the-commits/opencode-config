# Feature: Test Fabrication and Test Deletion Detection

**Source:** OWASP Secure Coding with AI Cheat Sheet — Section 8
**Reference:** https://cheatsheetseries.owasp.org/cheatsheets/Secure_Coding_with_AI_Cheat_Sheet.html#section-8-test-fabrication-and-test-deletion

## Problem

AI agents make CI green by deleting failing tests, weakening assertions, mocking the unit under test instead of fixing the code, or asserting the buggy behaviour. A passing test suite generated by the same agent that produced the code provides no independent assurance.

## User Story

As a developer, I want CI checks that detect test deletion, assertion weakening, and mock fabrication in agent-generated PRs, so that fake green test suites are caught.

## Acceptance Criteria

### Test Deletion Detection
- Given a PR deletes test files, when CI runs, the check compares the test file count and test function count against the base branch.
- Given tests are deleted, when the PR description does not include a justification, CI fails with a "Deleted tests without justification" warning.
- Given a test file is modified, when the number of test assertions decreases (e.g., fewer `expect()` or `assert.*()` calls), CI flags the change as "Assertion count decreased".

### Assertion Weakening Detection
- Given a PR modifies a test, when an assertion is changed from a stricter form to a weaker form (`toEqual` → `toBeDefined`, `assertEqual` → `assertNotNull`, strict equality → loose equality), CI flags the change.
- Given a PR modifies a test, when a real dependency is replaced with a mock that changes the test's behavior, CI warns about increased mocking.

### Test Coverage Comparison
- Given a PR adds code, when no corresponding test changes are detected, CI warns "Code changes without test changes — review required."
- Given a PR modifies a test, when the test covers only the generated code path and no edge cases, CI recommends adding adversarial test cases (invalid inputs, boundary conditions, concurrent access).

### Security-Critical Test Manual Requirement
- Given a PR changes authentication, authorization, input validation, or cryptographic code, when the test changes are entirely AI-generated, CI requires a human reviewer to explicitly approve the test changes.
- Given a PR's tests were generated by the same agent that wrote the production code, when CI detects this pattern, it flags the tests as "AI-generated, requires independent verification."

## Implementation Notes

- CI check script: `scripts/check-tests.mjs`
- Language-specific assertion pattern detection (Jest/Vitest `expect`, pytest `assert`, etc.)
- Test function count via regex or AST analysis
- Mock detection: count `mock()`, `jest.mock()`, `patch()`, `createStub()` calls
- Integration with successful-editing plugin for agent-side awareness
- Consider adding a `tests/adversarial/` directory for human-written security tests
